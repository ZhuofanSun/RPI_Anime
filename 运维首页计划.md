# 运维首页计划

## 目标

做一个统一的运维入口页，解决这几件事：

- 不再记 `Jellyfin`、`qBittorrent`、`AutoBangumi` 的端口号
- 在一个页面里看到服务是否在线、树莓派基础状态、下载链路状态
- 能看到 `manual_review` 队列积压情况
- 后续能通过页面处理人工审核内容
- 整个入口页只服务于自用环境，默认走局域网或 `Tailscale`

## 范围

这一阶段只做运维和入口，不做：

- 播放兼容性和转码优化
- 公网暴露或第三方登录
- 完整的多用户后台
- 全量自研 dashboard

## 方案结论

采用混合方案：

- `Homepage` 负责首页、入口、基础状态、服务卡片
- 自定义 `ops-review` 服务负责 `manual_review` 审核页

不建议现在自研整套前后端 dashboard，原因是：

- 入口、服务状态、系统状态这类能力，`Homepage` 已经够用
- 你真正需要自定义的部分，只有 `manual_review` 审核流
- 这样能把代码量和维护成本压到最低

## 组件职责

### Homepage

负责：

- 服务入口导航
- 服务基础状态展示
- 树莓派基础资源展示
- 下载链路相关只读概览
- 跳转到 `ops-review`

不负责：

- 直接处理审核动作
- 修改后处理规则
- 调用 `Tailscale` 官方云 API

### ops-review

建议技术栈：

- 后端：`FastAPI`
- 前端：服务端模板，优先 `Jinja2 + HTMX`

负责：

- 列出 `manual_review` 中待处理文件
- 展示文件路径、推断番名、季集信息、失败原因
- 提供人工操作：
  - 发布到 `Seasonal`
  - 删除
  - 重试解析
  - 后续扩展 `add mapping`

不负责：

- 替代 Jellyfin
- 替代 qBittorrent / AutoBangumi 原生 UI

### postprocessor

继续负责：

- 下载完成后的选优
- 发布到 `Seasonal`
- 生成 `tvshow.nfo` 和分集 `.nfo`
- 把未解析文件送入 `manual_review`

## 信息架构

运维首页建议分成 5 个区块。

### 1. 服务入口

固定放：

- Jellyfin
- qBittorrent
- AutoBangumi
- ops-review
- 项目 README 或运维文档入口

每个入口卡片显示：

- 名称
- 地址
- 在线状态
- 一句说明

### 2. 系统状态

展示树莓派基础健康信息：

- CPU 使用率
- 内存使用率
- 根分区使用率
- `/srv/anime-data` 使用率
- 温度
- 容器运行数量

### 3. 下载链路状态

展示：

- qB 当前下载数
- 当前上传数
- 下载速率
- 上传速率
- `Bangumi` 分类任务数
- `manual_review` 文件数
- 最近一次 postprocessor 处理状态

### 4. 网络状态

展示：

- 本机主机名
- 本机局域网地址
- `Tailscale` IP
- `Tailscale` 是否在线
- 最近在线 peer 数量

这里明确只走本地状态，不走官方云 API。

建议来源：

- `tailscale status --json`
- 或 `tailscale ip -4`

### 5. 快捷操作

第一阶段只放链接，不做危险按钮：

- 打开 `ops-review`
- 查看 postprocessor 日志
- 查看 compose 服务状态

真正有副作用的操作，比如“删除文件”“重试发布”，放到 `ops-review` 页面里，不放首页。

## 数据来源

### Homepage 直接读取

- `Docker` 容器状态
- `Jellyfin` widget
- `qBittorrent` widget
- 树莓派基础系统指标

### ops-review / 本地状态接口提供

- `manual_review` 文件列表和计数
- `postprocessor` 最近处理记录
- `Tailscale` 本地状态
- 目录容量和残留检查

## 权限和安全边界

默认原则：

- 运维首页只在局域网和 `Tailscale` 内访问
- 不单独暴露到公网
- 首页尽量只读
- 真正会改文件的操作只在 `ops-review` 中提供

建议：

- `Homepage` 使用最小权限配置
- `ops-review` 的文件操作全部限定在：
  - `/srv/anime-data/processing/manual_review`
  - `/srv/anime-data/library/seasonal`
  - `/srv/anime-data/downloads/Bangumi`

## 部署建议

### 服务新增

建议在 `compose.yaml` 里新增：

- `homepage`
- 之后再新增 `ops-review`

### 访问方式

建议用一个固定端口，例如：

- `Homepage`: `http://sunzhuofan.local:3000`

首页里统一跳转：

- `Jellyfin`
- `qBittorrent`
- `AutoBangumi`
- `ops-review`

这样以后只需要记一个入口。

## 分阶段落地

### 阶段 1：Homepage 基础版

目标：

- 先把入口和状态集中到一个页面

要做的事：

- 在 `compose.yaml` 里加入 `Homepage`
- 配好服务入口卡片
- 接好 `Jellyfin`、`qBittorrent`、`Docker`、系统状态 widget
- 首页加上 `AutoBangumi`、`ops-review` 占位入口

检查点：

- 可以通过一个固定地址打开首页
- 首页能跳转到 3 个现有服务
- 首页能看到容器和树莓派基础状态

完成标准：

- 不再需要手记服务端口

### 阶段 2：本地状态接口

目标：

- 把项目特有状态接到首页

要做的事：

- 提供一个很小的本地状态接口
- 输出：
  - `manual_review` 文件数
  - `Seasonal` 文件数
  - `Bangumi` 下载区残留数
  - `Tailscale` 本地状态摘要
  - 最近一次 postprocessor 处理结果

检查点：

- 首页可看到 `manual_review` 数量
- 首页可看到 `Tailscale` 在线状态和 tailnet IP

完成标准：

- 项目链路关键状态不需要再靠 SSH 查看

### 阶段 3：ops-review 只读页

目标：

- 先把人工审核内容可视化

要做的事：

- 增加 `ops-review` 服务
- 列表展示 `manual_review` 文件
- 展示：
  - 文件名
  - 路径
  - 推断标题
  - 季号 / 集号
  - 失败原因

检查点：

- 能从首页进入审核页
- 能在浏览器里看到待审核条目

完成标准：

- 不再需要手工 `find` 目录看审核内容

### 阶段 4：ops-review 可操作版

目标：

- 把常见人工处理动作网页化

要做的事：

- 增加按钮：
  - 发布到 `Seasonal`
  - 删除
  - 重试解析
- 记录操作结果
- 限制危险操作范围

检查点：

- 一条待审核内容可以在页面里完成处理
- 处理后首页数字会同步变化

完成标准：

- 日常运维不再依赖 SSH 手工操作

## 当前建议的下一步

当前建议先做 **阶段 1：Homepage 基础版**。

原因：

- 成本最低
- 立刻解决“端口号和入口分散”的问题
- 不会影响现有下载链路
- 后续可以自然承接 `ops-review`

阶段 1 做完后，再决定是否继续做阶段 2 和阶段 3。
