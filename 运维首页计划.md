# 运维首页计划

## 目标

做一个统一的运维入口页，解决这几件事：

- 不再记 `Jellyfin`、`qBittorrent`、`AutoBangumi` 的端口号
- 在一个页面里看到服务是否在线、树莓派基础状态、下载链路状态
- 能看到 `manual_review` 队列积压情况
- 后续能通过页面处理人工审核内容
- 整个入口页只服务于自用环境，默认走局域网或 `Tailscale`

## 范围

这一阶段只做运维和入口，不做：

- 播放兼容性和转码优化
- 公网暴露或第三方登录
- 完整的多用户后台
- 全量自研 dashboard

## 方案结论

采用自定义前后端 + 现成监控数据源的方案：

- `ops-ui` 负责首页、入口、基础状态、服务卡片
- `Glances` 继续负责系统和容器监控采集
- 自定义 `ops-review` 服务负责 `manual_review` 审核页

不再继续使用第三方 `Homepage`，原因是：

- 这一页已经不只是书签导航，而是项目专属控制台
- `Tailscale` 本地状态、下载残留、`manual_review` 等都不是现成卡片能稳定表达的
- 第三方 Homepage 的默认行为和配置约束反而增加了不稳定性
- 直接写静态前端 + 小型后端，维护边界反而更清晰

## 组件职责

### ops-ui

负责：

- 服务入口导航
- 服务基础状态展示
- 树莓派基础资源展示
- 下载链路相关只读概览
- 跳转到 `ops-review`
- 内部工具页的同站内多页面工作流

不负责：

- 直接处理审核动作
- 修改后处理规则
- 调用 `Tailscale` 官方云 API
- 替代 `Glances` 采集底层监控

### ops-review

建议技术栈：

- 后端：`FastAPI`
- 前端：服务端模板，优先 `Jinja2 + HTMX`

负责：

- 列出 `manual_review` 中待处理文件
- 展示文件路径、推断番名、季集信息、失败原因
- 提供人工操作：
  - 发布到 `Seasonal`
  - 删除
  - 重试解析
  - 后续扩展 `add mapping`

不负责：

- 替代 Jellyfin
- 替代 qBittorrent / AutoBangumi 原生 UI

### postprocessor

继续负责：

- 下载完成后的选优
- 发布到 `Seasonal`
- 生成 `tvshow.nfo` 和分集 `.nfo`
- 把未解析文件送入 `manual_review`

## 信息架构

运维首页建议分成 5 个区块。

## 导航约定

采用混合导航：

- 外部服务：
  - `Jellyfin`
  - `qBittorrent`
  - `AutoBangumi`
  - `Glances`
  继续新标签页打开，避免打断运维台上下文。
- 内部工作页：
  - `Postprocessor`
  - `Ops Review`
  - `Logs`
  - `Tailscale`
  统一放在 `ops-ui` 站内，用多页面跳转、返回按钮和共享样式。
  首页和内部工作页优先使用首屏骨架与会话缓存，避免每次切页都先看到空白面板。

### 1. 服务入口

固定放：

- Jellyfin
- qBittorrent
- AutoBangumi
- ops-review
- 项目 README 或运维文档入口

每个入口卡片显示：

- 名称
- 地址
- 在线状态
- 一句说明

### 2. 系统状态

展示树莓派基础健康信息：

- CPU 使用率
- 内存使用率
- 根分区使用率
- `/srv/anime-data` 使用率
- 温度
- 容器运行数量
- 24 小时 CPU / 温度趋势
- 近 7 日下载量柱状图

### 3. 下载链路状态

展示：

- qB 当前下载数
- 当前上传数
- 下载速率
- 上传速率
- `Bangumi` 分类任务数
- `manual_review` 文件数
- 最近一次 postprocessor 处理状态

### 4. 网络状态

展示：

- 本机主机名
- 本机局域网地址
- `Tailscale` IP
- `Tailscale` 是否在线
- 最近在线 peer 数量

这里明确只走本地状态，不走官方云 API。

建议来源：

- `tailscale status --json`
- 或 `tailscale ip -4`

### 5. 快捷操作

第一阶段只放链接，不做危险按钮：

- 打开 `ops-review`
- 查看 postprocessor 日志
- 查看 compose 服务状态

真正有副作用的操作，比如“删除文件”“重试发布”，放到 `ops-review` 页面里，不放首页。

## 数据来源

### ops-ui 后端聚合

- `Glances API`：
  - CPU
  - 内存
  - 容器状态
- `qBittorrent Web API`：
  - `Bangumi` 分类任务数
  - 下载/上传速率
  - 活跃下载/做种数
- 本地文件系统：
  - `manual_review` 文件数
  - `Seasonal` 文件数
  - 下载区残留数
- 本地 `Tailscale` socket：
  - `Tailscale` 在线状态
  - tailnet IP
  - peer 数量

### ops-review

- `manual_review` 文件列表和详情
- 人工审核操作结果
- 之后再加 `postprocessor` 最近处理记录

## 权限和安全边界

默认原则：

- 运维首页只在局域网和 `Tailscale` 内访问
- 不单独暴露到公网
- 首页尽量只读
- 真正会改文件的操作只在 `ops-review` 中提供

建议：

- `ops-ui` 只暴露只读聚合接口
- `ops-review` 的文件操作全部限定在：
  - `/srv/anime-data/processing/manual_review`
  - `/srv/anime-data/library/seasonal`
  - `/srv/anime-data/downloads/Bangumi`

## 部署建议

### 服务新增

建议在 `compose.yaml` 里新增：

- `ops-ui`
- 之后再新增 `ops-review`

### 访问方式

建议用一个固定端口，例如：

- `Ops UI`: `http://sunzhuofan.local:3000`

首页里统一跳转：

- `Jellyfin`
- `qBittorrent`
- `AutoBangumi`
- `ops-review`

这样以后只需要记一个入口。

## 分阶段落地

### 阶段 1：ops-ui 基础版

目标：

- 先把入口和状态集中到一个页面

要做的事：

- 在 `compose.yaml` 里加入 `ops-ui`
- 用 `HTML + CSS + JS` 写服务入口卡片
- 用 `FastAPI` 提供聚合状态接口
- 接好 `Glances`、`qBittorrent`、本地 `Tailscale` 状态
- 首页加上 `AutoBangumi`、`ops-review` 占位入口

检查点：

- 可以通过一个固定地址打开首页
- 首页能跳转到 3 个现有服务
- 首页能看到容器和树莓派基础状态

完成标准：

- 不再需要手记服务端口

### 阶段 1A：交互与布局修正

目标：

- 先把当前首页的基础交互顺手度修好

要做的事：

- 所有 `Open Service` 默认在新标签页打开
- 规避 `qBittorrent` 同页打开时的 `Unauthorized` 体验问题
- 让服务入口区块从“7 张卡片不平衡”调整为更稳定的布局
- 服务入口固定成 8 张卡片
- 增加第 8 张可打开的卡片，优先是日志页占位
- 给首页增加显式的“上次刷新时间”和后续可配置刷新间隔入口

检查点：

- 所有可打开服务都默认新标签页打开
- `qBittorrent` 不再因为同页跳转带来糟糕体验
- 服务卡片区块的视觉重心不再偏一侧

完成标准：

- 首页的基础导航交互可长期使用，不需要靠浏览器后退反复切页

### 阶段 1B：主机状态增强

目标：

- 让主机和服务卡片更像真正的运维面板，而不是只显示几个基础数字

要做的事：

- 增加树莓派主机运行时长
- 增加服务运行时长
- 增加 CPU 当前温度
- 评估并增加 CPU 平均温度或一段时间窗口平均值
- 增加系统负载信息，例如 `load average`
- 明确在线 / 离线 / 启动中 / 异常几种状态的来源和显示规则
- 把当前的自动刷新频率配置化
- 增加第一批小图，优先用轻量 sparkline 展示 CPU、温度、下载和上传趋势

补充说明：

- 当前卡片会自动刷新，前端默认每 `8` 秒请求一次 `/api/overview`
- 当前在线 / 离线状态主要来自容器状态和本地 `Tailscale` 状态
- 主机信息还偏少，这一阶段补齐

检查点：

- 可以直接在首页看到树莓派已经开机多久
- 可以直接看到各服务已经运行多久
- CPU 温度和负载可读，不需要另开 `Glances`

完成标准：

- 常见“机器现在健康不健康”的判断不需要先 SSH 再查命令

### 阶段 1C：服务控制能力

目标：

- 在首页加入基础控制能力，但不把它做成高风险的全能后台

要做的事：

- 增加单个服务重启按钮
- 增加整套服务重启按钮
- 为危险操作增加确认和结果提示
- 给控制动作设计权限边界

实现建议：

- 不建议直接让前端调用宿主机命令
- 二选一：
  - 给 `ops-ui` 加一个受限操作后端，并授予它最小化的 Docker 控制能力
  - 单独做一个很小的受控 helper，只暴露“重启单服务 / 重启整套”这几个固定动作

检查点：

- 可以从首页重启单个服务
- 可以从首页重启整套服务
- 操作结果和失败原因能回显给用户

完成标准：

- 常见恢复动作不再依赖手工 SSH

### 阶段 1D：视觉系统升级

目标：

- 保留现在的浅色基调，同时把首页做得更有层次和更现代

要做的事：

- 增加 `light / dark` 切换
- 提高卡片和背景的对比，让信息块更立体
- 增加卡片悬浮效果
- 增加按钮悬浮和按下效果
- 给状态点、按钮和数字卡片补更清晰的视觉层级
- 避免过度扁平，适度增加阴影、边缘高光和渐变

检查点：

- 浅色和深色模式都可用
- 卡片在悬浮时有明确反馈
- 主要操作按钮在视觉上更明显

完成标准：

- 首页看起来像一个完整产品页面，而不是只读调试面板

### 阶段 2：项目状态细化

目标：

- 把项目特有状态接到首页

要做的事：

- 在 `ops-ui` 后端补项目特有聚合
- 输出：
  - `manual_review` 文件数
  - `Seasonal` 文件数
  - `Bangumi` 下载区残留数
  - `Tailscale` 本地状态摘要
  - 最近一次 `postprocessor` 处理结果

检查点：

- 首页可看到 `manual_review` 数量
- 首页可看到 `Tailscale` 在线状态和 tailnet IP

完成标准：

- 项目链路关键状态不需要再靠 SSH 查看

### 阶段 3：ops-review 只读页

目标：

- 先把人工审核内容可视化

要做的事：

- 增加 `ops-review` 服务
- 列表展示 `manual_review` 文件
- 展示：
  - 文件名
  - 路径
  - 推断标题
  - 季号 / 集号
  - 失败原因

检查点：

- 能从首页进入审核页
- 能在浏览器里看到待审核条目

完成标准：

- 不再需要手工 `find` 目录看审核内容

### 阶段 4：ops-review 可操作版

目标：

- 把常见人工处理动作网页化

要做的事：

- 增加按钮：
  - 发布到 `Seasonal`
  - 删除
  - 重试解析
- 记录操作结果
- 限制危险操作范围

检查点：

- 一条待审核内容可以在页面里完成处理
- 处理后首页数字会同步变化

完成标准：

- 日常运维不再依赖 SSH 手工操作

## 当前建议的下一步

当前建议先做 **阶段 1A：交互与布局修正**。

原因：

- 成本最低
- 直接修复当前已经暴露出来的使用问题
- 不会影响现有下载链路
- 做完后再进入主机状态增强和服务控制会更顺

推荐顺序：

1. `阶段 1A：交互与布局修正`
2. `阶段 1B：主机状态增强`
3. `阶段 1D：视觉系统升级`
4. `阶段 1C：服务控制能力`
5. `阶段 2 / 阶段 3 / 阶段 4`
