# 树莓派私人影音库方案

更新时间：2026-04-03

## 1. 目标与范围

你的目标比较明确：

- 纯自用。
- 主要是番剧订阅，不是电影库优先。
- 常见片源是 1080p，偶尔 1440p，码率不高。
- 不同网络环境下访问。
- 以自动订阅、自动下载、自动整理为主。
- 片源命名复杂，字幕组差异大，容器可能是 `mkv` 或 `mp4`。
- 字幕可能是外挂字幕、内封字幕，或者已经硬字幕压进视频。
- 可以接受“下载后离线处理”，不要求播放时实时转码。

这决定了方案应该优先满足以下原则：

- 优先保证 `Direct Play`，尽量不依赖边播边转码。
- 自动化链路要围绕“番剧文件整理”设计，而不是先围绕电影/美剧生态设计。
- 必须把“下载区”和“媒体库区”拆开，先处理再入库。
- 命名、字幕、容器兼容问题应该在入库前解决，而不是把脏数据直接丢给 Jellyfin。
- 树莓派 4B 更适合承担“下载、整理、刮削、轻量服务、离线批处理”，不适合承担持续高强度实时转码。

## 2. 当前硬件的适配结论

当前硬件：

- Raspberry Pi 4B 4GB
- 32GB SD 卡

结论：

- 这套硬件够做自用番剧库的“控制面”和“轻量数据面”。
- 不适合把所有文件长期跑复杂重编码，尤其不适合多人并发实时转码。
- `SD 卡只适合临时过渡`，不适合长期承受下载缓存、数据库、日志和容器写入。
- 应该尽快把系统迁到外置 SSD，并把媒体数据放到独立大盘。

建议的硬件落地：

- 系统盘：`1TB SSD`
- 媒体盘：`8TB` 起步的独立供电 HDD

推荐原因：

- 系统盘负责系统、Docker、数据库、下载缓存、转码临时目录。
- 媒体盘负责归档后的番剧库。
- 下载缓存和媒体盘在同一文件系统内时，可以使用 `hardlink` 降低重复占用。
- 如果你后面需要批量重封装或预转码，SSD 作为暂存区会稳定很多。

## 3. 推荐的软件栈

### 3.1 第一阶段最适合你的组合

- `Raspberry Pi OS Lite 64-bit`
- `Docker Compose`
- `Tailscale`
- `Jellyfin`
- `qBittorrent-nox`
- `AutoBangumi`
- `自定义后处理程序`
- 可选：`Unmanic`

### 3.2 各组件职责

`AutoBangumi`

- 负责番剧 RSS 订阅、规则匹配、提交下载任务。
- 更贴近“追番”场景，比一开始就上全套 Servarr 更轻。

`qBittorrent-nox`

- 负责下载。
- 适合树莓派这种无桌面环境。
- 与 AutoBangumi 配合做自动下载很自然。

`自定义后处理程序`

- 这是你方案里最值得单独加的一层。
- 负责统一命名、做流信息探测、处理字幕、做必要的重封装、触发离线预转码、最终入库。
- 这是解决“字幕组差异大、命名复杂不统一”的关键。

`Jellyfin`

- 负责媒体展示、元数据、播放和多端访问。
- 只吃“已经整理好的库”。
- 不直接读取下载目录。

`Tailscale`

- 负责跨网络访问。
- 纯自用优先走 Tailscale，不建议把 Jellyfin 端口直接裸露到公网。

`Unmanic`

- 适合做“轻量离线整理/转码/重封装”。
- 比 Tdarr 更简单，树莓派上更现实。
- 如果你只是想做少量统一化处理，优先 Unmanic。

### 3.3 为什么不建议一开始就上 Sonarr/Radarr/Prowlarr/Bazarr 全家桶

原因不是它们不好，而是它们不完全贴合你当前重点：

- 你的核心是“番剧订阅 + 字幕组差异处理 + 命名统一”。
- `AutoBangumi` 更接近这个问题。
- `Bazarr` 目前仍依赖 `Sonarr/Radarr`，不适合直接挂在 AutoBangumi 方案上。
- `Prowlarr + Sonarr + Radarr + Bazarr + Jellyseerr` 更像“综合影音库平台”，适合后续扩展，不适合作为第一步。

建议：

- 第一阶段先把番剧链路跑顺。
- 第二阶段如果你开始管电影、美剧、请求式入库，再扩展到 Servarr 体系。

## 4. 目录规划

建议从一开始就把路径规划好。

```text
/srv/media
├── downloads
│   ├── incomplete
│   ├── complete
│   └── watch
├── processing
│   ├── incoming
│   ├── working
│   ├── failed
│   └── reports
├── library
│   └── anime
├── archive
│   └── original
└── cache
    ├── transcode
    └── thumbnails
```

说明：

- `downloads/complete` 只表示“下载完成”，不表示可以直接给 Jellyfin 读取。
- `processing/incoming` 是后处理程序的入口。
- `processing/working` 用来做 `ffprobe`、重命名、重封装、字幕抽取、预转码。
- `library/anime` 是 Jellyfin 真正读取的目录。
- `archive/original` 可选，用于保留原始文件。
- `cache/transcode` 存放离线转码临时文件。

## 5. 入库链路设计

推荐的数据流：

```text
AutoBangumi
  -> qBittorrent 下载
  -> 下载完成回调
  -> 后处理程序接管
  -> 文件名解析
  -> ffprobe/mediainfo 探测
  -> 字幕/容器决策
  -> 重命名/重封装/必要时生成兼容版
  -> 移入或硬链接到媒体库
  -> 通知 Jellyfin 扫库
```

关键点：

- 不要让 Jellyfin 直接扫下载目录。
- 只有“处理完成”的文件才允许进入媒体库。
- 每个文件处理完都应该输出一份结构化报告，方便追溯。

建议输出的报告字段：

- 原始文件名
- 解析出的番名
- 集数
- 分辨率
- 视频编码
- 音频编码
- 字幕轨数量和类型
- 外挂字幕列表
- 最终文件名
- 最终库路径
- 是否做了重封装
- 是否做了转码
- 是否保留原始文件

## 5.1 订阅规则要尽量在源头降噪

对你这种追番场景，最省事的做法不是“什么都下回来再处理”，而是先把订阅规则写得更挑剔。

建议你在订阅阶段就建立这些偏好：

- 优先的字幕组白名单
- 明确拒绝的字幕组黑名单
- 优先 `1080p`
- 可接受 `1440p`
- 非必要不收过大的原盘流或高码率版本
- 优先 `mkv`
- 可接受 `mp4`
- 非必要先不收 `AV1`
- 如果你主要客户端不统一，先把 `10-bit HEVC` 也设成次优先级
- 尽量优先“内封字幕/外挂字幕”，降低硬字幕版本的优先级

如果 AutoBangumi 规则能表达这些偏好，就尽量在订阅层解决。

原因：

- 源头越干净，后处理越简单。
- 树莓派最不该做的事情就是替你反复处理本来可以不下的版本。
- 订阅阶段少犯错，比下载后补救便宜得多。

### 5.2 基于现有样本的实际规则建议

如果以你当前 `/Volumes/Anime` 里的样本作为参考，后处理规则可以明显收窄。

目前主要来源基本集中在这些模式：

- `ANi / Baha / 1080P / AVC / AAC / mp4`
- `Feibanyama / 2160p / HEVC / OPUS / Multi-Subs / mkv`
- `LoliHouse / 1080p / HEVC-10bit / AAC / SRTx2 / mkv`
- `SweetSub&VCB-Studio / mkv + 外挂 ass + 字体包`
- `Dynamis One / 1080p / AVC / AAC / mp4`
- 少量 `Nekomoe kissaten`
- 少量 `樱桃花字幕组` 简日内嵌版
- 少量 `DBD-Raws` 剧场版、特典、菜单、PV 等大包

这意味着第一版规则不需要追求“万能”。

建议直接写成几条显式规则：

`ANi / Dynamis One`

- 基本视为轻量稳定源。
- 通常只做重命名。
- `mp4` 可以保留，不必强制转 `mkv`。

`Feibanyama`

- 默认优先级降低。
- 你现有样本里这一类大量是 `2160p HEVC OPUS Multi-Subs`。
- 如果你未来订阅目标是“季度追番看完就删”，这类版本很容易吃掉空间，也更容易触发客户端兼容问题。
- 建议能选 `1080p` 就不要收 `2160p`。

`LoliHouse`

- 作为高质量来源保留。
- 一般只做规范命名。
- 如果字幕是外挂 `SRTx2`，可直接作为兼容版源。

`SweetSub&VCB-Studio`

- 作为“收藏型/高质量型”来源保留。
- 默认保留原始 `ASS` 和字体包。
- 这类更适合进入“保真库”，不适合默认做强制兼容化。

`樱桃花字幕组` 内嵌字幕版

- 如果你已经有更高质量或更规范来源，应降低优先级。
- 这类文件更适合“缺集补位”，不适合作为长期统一标准。

`DBD-Raws` 大包

- 默认进入 `manual_review`。
- 不自动入库。
- 不自动保留特典、PV、NCOP、菜单目录。

### 5.3 基于现有样本的过滤规则

你已经明确提到 `物语` 这类大型收藏目录应该忽略，这个思路是对的。

建议在订阅和入库两层都加过滤：

订阅层过滤：

- 关键字命中 `特典`
- 关键字命中 `SP`
- 关键字命中 `PV`
- 关键字命中 `NCOP`
- 关键字命中 `NCED`
- 关键字命中 `menu`
- 关键字命中 `Fonts`
- 命中你定义的“收藏型长期系列黑名单”，例如 `物语`

入库层过滤：

- 目录内不止单集文件，还包含 `PV`、`NCOP`、`menu`、`Scans`、`CDs`
- 单个任务体积异常大
- 同一任务包含大量非正片资源
- 无法明确判断是否为“单周更新的一集”

对于这些情况，建议直接：

- 不自动发布到主媒体库
- 标记为 `manual_review`
- 或丢到单独的 `archive/raw-package`

## 6. 命名与元数据统一

这是你场景里最重要的部分。

### 6.1 Jellyfin 能不能直接处理这些乱命名

可以处理一部分，但不应该把它当成“最终清洗器”。

Jellyfin 支持常见 `mp4` 和 `mkv` 容器，也支持外挂字幕和内封字幕，但它依赖相对规范的目录结构和文件命名来稳定识别媒体。文件名里如果包含保留字符，或者番名、季、集信息表达得太混乱，识别质量会明显下降。

所以建议：

- 入库前统一命名。
- 必要时生成 `.nfo` 辅助元数据匹配。
- 不要把字幕组原始长标题直接作为最终库文件名。

### 6.2 推荐的番剧命名格式

建议统一成：

```text
/srv/media/library/anime/
└── 番剧名
    └── Season 01
        ├── 番剧名 S01E01.mkv
        ├── 番剧名 S01E02.mkv
        └── 番剧名 S01E03.mkv
```

如果同一集要保留外挂字幕：

```text
番剧名 S01E01.mkv
番剧名 S01E01.zh-Hans.ass
番剧名 S01E01.zh-Hant.ass
番剧名 S01E01.ja.ass
番剧名 S01E01.en.srt
```

建议命名只保留：

- 标准番名
- 季号
- 集号
- 可选标题

不要保留：

- 字幕组名
- 分发站点标签
- 哈希串
- 冗长编码说明
- 无意义宣传字段

这些信息如果你想保留，应该进数据库或报告文件，而不是进最终库文件名。

### 6.3 推荐后处理程序怎么实现

建议程序拆成 4 步：

`1. 文件名解析`

- 优先用文件名猜测番名、集号、分辨率、字幕组。
- 可考虑 `anitopy` 这类 anime 文件名解析库。
- 对无法稳定识别的番剧维护一个手工映射表。

`2. 媒体探测`

- 用 `ffprobe` 或 `mediainfo` 探测：
  - 容器类型
  - 视频编码
  - 音频轨数量和语言
  - 字幕轨数量、格式、默认标记、强制标记
  - 是否存在附件字体

`3. 规则决策`

- 根据规则判断：
  - 直接入库
  - 仅重命名
  - 重封装为 `mkv`
  - 提取外挂字幕
  - 生成兼容版字幕
  - 进入离线预转码队列

`4. 发布入库`

- 成功后原子移动到最终目录。
- 失败则放入 `processing/failed`。
- 成功后调用 Jellyfin API 触发指定目录刷新。

### 6.4 建议优先做“重封装”而不是“重编码”

对于你这种番剧源：

- `mp4` 和 `mkv` 混用本身不是最大问题。
- 真正的问题通常是字幕轨、附件字体、默认音轨、默认字幕轨、命名和元数据混乱。

因此优先级应该是：

1. 规范命名
2. 规范目录
3. 统一默认轨
4. 必要时把 `mp4` 重封装成 `mkv`
5. 只在确有必要时才重编码

原因：

- `mkv` 对多音轨、多字幕、附件字体的承载更友好。
- 对 anime 来说，很多字幕组的 `ASS` 样式、字体附件和章节信息，放在 `mkv` 里更稳。
- 单纯 remux 几乎不损失画质，资源消耗也远低于重编码。

## 7. 字幕处理策略

### 7.1 先把字幕分成 3 类

`1. 硬字幕`

- 字幕已经压进画面。
- 无法关闭。
- 这种文件不需要额外处理字幕轨，但也无法做字幕切换。

`2. 内封字幕`

- 字幕轨存在于容器内，可以开关。
- 常见于 `mkv`。
- 可能带 `ASS` 样式，也可能是 `SRT`。

`3. 外挂字幕`

- 与视频同名的 `.ass`、`.srt`、`.vtt` 等文件。
- Jellyfin 可以识别规范命名的外挂字幕。

### 7.2 对 anime 的建议

优先策略：

- 保留原始 `ASS` 字幕。
- 不要默认把 `ASS` 全部转换成 `SRT`。
- 不要默认把字幕烧进视频。

原因：

- anime 字幕组常见的特效、注释、路牌翻译、karaoke、字体依赖，很多都依赖 `ASS`。
- 转成 `SRT` 会丢失样式、定位、特效和部分信息。
- 烧录字幕会让文件失去可切换性，而且会大幅提高处理成本。

### 7.3 什么时候才建议额外生成 SRT 兼容字幕

仅在以下情况考虑：

- 你主要用浏览器播放。
- 你经常在移动网络下从 Web 端看。
- 某些客户端对 `ASS` 支持不好，导致 Jellyfin 需要转码或烧字幕。

更合理的做法不是“替换掉 ASS”，而是：

- 保留原始 `ASS`
- 额外生成一个简化版 `SRT` 作为兼容字幕

这样你有两个播放路径：

- 支持好的客户端，读原始 `ASS`
- 支持差的客户端，手动切换到 `SRT`

注意：

- `ASS -> SRT` 是降级兼容，不是无损转换。
- 如果字幕 heavily typeset，转换结果可能很差。

### 7.4 是否需要 Bazarr

当前阶段不建议把它作为核心组件。

原因：

- 你的主要来源是番剧片源，本身往往已带字幕。
- 你现在的核心问题是“字幕轨混乱和文件整理”，不是“补抓缺失字幕”。
- Bazarr 依赖 Sonarr/Radarr，不适合直接挂在 AutoBangumi 方案上。

它更适合未来你转向综合影音库后再接入。

## 8. 离线预转码方案

你提的这个方向是对的，而且比实时转码更适合 Pi 4。

### 8.1 原则

- 不要把树莓派当实时转码机。
- 可以把它当“空闲时慢慢处理文件的离线整理机”。

推荐策略：

- 下载完成后，先判断是否真的需要预转码。
- 对不需要处理的文件直接入库。
- 对需要处理的文件加入离线队列，空闲时慢慢做。

### 8.2 哪些情况适合离线预处理

`适合 remux`

- `mp4` 改成 `mkv`
- 调整默认音轨/字幕轨
- 提取外挂字幕
- 清理多余元数据

`适合轻量转码`

- 某些冷门编码导致客户端兼容差
- 音频编码过于奇怪，需要转成 `AAC`
- 字幕轨无法被目标客户端稳定处理，需要额外生成兼容版本

`不建议默认转码`

- 已经是常见编码，且客户端能直放
- anime 原始片源依赖复杂 `ASS` 样式
- 画质本来就不高，再压只会继续损失

### 8.3 你最实际的预转码目标

你这个场景下，真正有价值的不是“把所有片子都压一遍”，而是建立一个兼容优先的目标格式。

建议优先目标：

- 容器：`mkv`
- 视频：优先保留原视频，不强制重编码
- 音频：不兼容时转 `AAC`
- 字幕：保留原始 `ASS`，必要时补一份 `SRT`

如果你明确知道主要客户端都支持 `H.264 + AAC + SRT`，那么可以加一个“兼容版”目标：

- 容器：`mp4` 或 `mkv`
- 视频：`H.264 8-bit`
- 音频：`AAC`
- 字幕：外挂 `SRT`

但这更像“额外导出一个通用副本”，不适合拿来替代所有 anime 原始文件。

### 8.4 为什么不建议把所有番剧统一压成一个格式

主要问题有 4 个：

- Pi 4 编码性能有限，批量重编码时间很长。
- 二次压制有画质损失。
- anime 的字幕、字体、章节和样式经常不是“只转个码就完事”。
- 你当前片源本身是 1080p 低码率，继续压节省的空间不一定值得付出的时间和复杂度。

更好的路线是：

- 先做“命名统一 + 容器统一 + 字幕整理”
- 真遇到兼容问题，再针对性离线转码

### 8.5 推荐的离线处理工具选择

优先级建议：

`方案 A：自定义脚本 + ffmpeg/mkvtoolnix`

- 最适合你这种“规则明确、愿意自己控制”的场景。
- 足够轻。
- 可完全围绕 anime 规则设计。

`方案 B：Unmanic`

- 适合在脚本之外补一层自动检测和排队处理。
- 更容易做“发现不符合规则的文件就处理”。
- 比 Tdarr 更简单。

`方案 C：Tdarr`

- 功能最强，插件化强，能做非常复杂的流水线。
- 但对你当前机器和需求来说，明显偏重。
- 后期如果你把 Pi 只做控制节点，把转码丢给更强的机器，Tdarr 才更有价值。

### 8.6 离线预转码的实际落地建议

建议给后处理程序加一个策略层：

```text
if 文件可直接被主力客户端稳定直放:
    直接入库
elif 仅容器/字幕/音轨存在兼容问题:
    做 remux / 提取字幕 / 生成兼容字幕
elif 只有特定设备不兼容:
    生成兼容副本
else:
    标记人工检查
```

这比“一律压一遍”更合理。

### 8.7 自定义后处理程序的推荐实现

如果这层你准备自己写，我建议按“规则引擎 + 队列 + 工具调用”的思路实现，而不是一上来做成复杂 Web 平台。

推荐技术栈：

- `Python 3.11+`
- `SQLite`
- `ffprobe`
- `ffmpeg`
- `mkvmerge`
- `mkvpropedit`
- `mediainfo`

为什么这样选：

- Python 足够快，适合做文件编排和规则判断。
- SQLite 足够支撑单机任务状态，不必上 PostgreSQL。
- `ffprobe` 负责事实探测。
- `mkvmerge` / `mkvpropedit` 比直接拿 `ffmpeg` 硬改轨道元数据更稳。
- `ffmpeg` 只做确实需要的重封装、抽轨和转码。

建议程序模块：

- `watcher`
- `parser`
- `probe`
- `decision_engine`
- `executor`
- `publisher`
- `reporter`

各模块职责：

`watcher`

- 监听下载完成目录或消费下载完成回调。

`parser`

- 解析文件名，识别番名、集号、字幕组、分辨率。
- 维护人工覆盖映射表。

`probe`

- 读取视频、音频、字幕、附件字体、章节信息。

`decision_engine`

- 根据规则输出动作清单。

`executor`

- 实际执行 rename、remux、字幕抽取、生成兼容字幕、转码。

`publisher`

- 把结果移动或硬链接到媒体库。
- 刷新 Jellyfin。

`reporter`

- 写入 SQLite 和 JSON 报告，方便复查。

建议状态机：

- `downloaded`
- `parsed`
- `probed`
- `planned`
- `processing`
- `published`
- `failed`
- `manual_review`

建议每个任务都保存：

- 输入路径
- 输出路径
- 决策理由
- 执行命令
- 执行耗时
- 错误信息
- 媒体指纹

### 8.8 推荐的判定规则边界

建议你把规则写成“先保守，再逐步放开”的形式。

第一批规则建议只有这些：

- 文件名能识别出番名和集号，才允许自动入库。
- 识别失败直接进 `manual_review`。
- `mkv` 且视频编码常见时，只做重命名和默认轨设置。
- `mp4` 且带多字幕/多音轨时，优先 remux 到 `mkv`。
- 字幕为 `ASS` 时默认保留，不自动转 `SRT`。
- 外挂字幕存在且命名不规范时，只改名，不改内容。
- 如果存在多个中文字幕轨，只设一个默认轨，其余保留。
- 如果音频轨不是常见格式且客户端兼容性差，再转 `AAC`。

不建议一开始就自动做的事情：

- OCR 图片字幕
- 全量 `ASS -> SRT`
- 全量重编码
- 自动删除原始文件
- 自动覆盖已入库同名文件

### 8.9 关于图片字幕和 OCR 的结论

如果遇到的是：

- `PGS`
- `VobSub`
- 其他图片型字幕

我的建议是：

- 第一阶段不要在树莓派上做自动 OCR。
- 先原样保留，标记为兼容风险。
- 只有你确认某类片源经常造成播放问题，再单独对这类片源做 OCR 流程。

原因：

- OCR 本身耗时。
- 动漫场景里 OCR 质量不稳定。
- 一旦自动化失败，很容易得到时间轴错乱或文本错误的字幕。

## 9. 客户端与播放策略

如果你想减少转码需求，客户端选择非常关键。

优先推荐：

- Jellyfin 官方客户端
- Android TV 客户端
- iOS 上的 Infuse
- 桌面端优先专用客户端，不优先浏览器

原因：

- 浏览器在编码和字幕支持上通常更保守。
- Jellyfin 官方文档也明确说明，字幕会导致 Direct Stream 或视频转码，烧字幕尤其吃资源。
- `H.264 8-bit` 的通用性最好，但 `ASS` 字幕和复杂字幕渲染不一定在所有 Web 客户端都表现一致。

结论：

- 如果你主要用专用客户端，保留原始字幕方案更现实。
- 如果你主要用浏览器，才更需要“兼容字幕”或“兼容副本”。

## 10. 外网访问方案

纯自用优先：

- `Tailscale`

理由：

- 部署简单。
- 不依赖公网 IP。
- 不需要自己维护证书、反代和暴露服务面。

建议：

- Jellyfin 仅绑定内网和 Tailscale 网络。
- 管理面板不要直接暴露公网。
- 给长期在线节点关闭 key expiry，避免路由意外失效。

不建议当前阶段优先做：

- 直接公网端口映射 Jellyfin
- 裸暴露 qBittorrent / AutoBangumi

如果后面你明确需要公网域名访问，再加：

- `Caddy`
- HTTPS
- 严格认证

## 11. 存储建议

### 11.1 最推荐的组合

`系统盘`

- `1TB SSD`
- 装系统、Docker、数据库、下载缓存、工作目录

`媒体盘`

- `8TB` 独立供电 HDD

这个组合适合：

- 成本合理
- 稳定
- 方便未来升级

### 11.2 如果你要更进一步

当容量超过单盘舒适区后，再考虑：

- 双盘镜像
- 多盘 DAS
- 定期冷备份

但要注意：

- `RAID != 备份`
- 影音库即使可以重下，配置、元数据、处理规则和映射表也很值得备份

最该备份的不是视频本身，而是：

- Docker Compose
- 配置文件
- AutoBangumi 规则
- Jellyfin 配置
- 你自定义后处理脚本
- 番名映射表
- 处理报告数据库

### 11.3 你的简化硬件方案也可以成立

如果按你最新约束：

- 不要 HDD
- 一个季度删一次本地资源
- `500GB` 外置移动盘足够
- 项目想继续放在 `SD` 卡上

那可以改成更轻的方案：

- `SD 卡`：系统和极少量只读配置
- `500GB 外置 SSD`：下载、处理、媒体库、Jellyfin 数据、qBittorrent 数据、日志、缓存

这个前提下，关键不是“能不能继续用 SD”，而是“不要把高写入目录继续留在 SD 上”。

我不建议放在 SD 上的内容：

- `qBittorrent` 下载目录
- `qBittorrent` 数据库与会话状态
- `Jellyfin` 元数据和数据库
- 转码缓存
- 自定义后处理程序的 SQLite 和报告
- Docker 的可写卷

可以放在 SD 上的内容：

- Raspberry Pi OS 本体
- 少量静态配置
- 引导和基础服务

结论：

- `系统留在 SD` 可以接受。
- `业务数据和所有高写入目录` 应该放外置 SSD。
- 如果你把 Docker 数据目录、下载目录、缓存目录全留在 SD，长期稳定性会明显变差。

### 11.4 容量上 500GB 对你当前模式基本够用

基于你现在 `/Volumes/Anime` 的实际样本：

- 全部目录合计约 `161GB`
- 排除你明确说要忽略的 `物语` 后，约 `97GB`

这说明在“季度追番、看完清理、不长期囤大包”的模式下：

- `500GB` 外置 SSD 完全够用
- 甚至还能保留一部分缓存、失败队列和短期归档

但有一个前提：

- 订阅阶段尽量少收 `2160p` 大体积版本

因为你当前样本里已经有相当一批 `Feibanyama 2160p HEVC OPUS Multi-Subs`，这类文件是最容易把空间吃掉的一类。

## 12. 运维与监控

至少要做这些：

- 定期检查磁盘 SMART
- 检查下载区残留
- 检查 Jellyfin 扫库失败日志
- 检查转码/重封装失败队列
- 监控 SSD 空间和媒体盘空间
- 监控 CPU 温度

建议做法：

- 每周一次健康检查脚本
- 每天一次清理脚本
- 对 `processing/failed` 做通知

## 13. 最推荐的实施顺序

### 阶段 1：先把基础打稳

- SSD 启动
- Docker Compose
- Tailscale
- Jellyfin
- qBittorrent-nox
- AutoBangumi

目标：

- 能下载
- 能入库
- 能在外部网络访问

### 阶段 2：补上后处理程序

- 文件名解析
- ffprobe 探测
- 统一命名
- 规范目录
- 简单 remux
- Jellyfin API 刷库

目标：

- 让媒体库输入变干净

### 阶段 3：再加离线兼容处理

- 只对特定规则触发预处理
- 生成兼容字幕或兼容副本
- 增加人工复核标记

目标：

- 降低实际播放时的转码概率

### 阶段 4：最后才考虑扩展生态

- Sonarr/Radarr/Prowlarr
- Bazarr
- 更强服务器承担转码

## 14. 对你当前问题的最终判断

你的重点不是“Jellyfin 行不行”，而是“Jellyfin 前面的清洗流程是否足够干净”。

结论：

- Jellyfin 可以作为最终前端。
- 真正决定体验的是入库前的那层后处理。
- 对你的片源类型，最有价值的是：
  - 统一命名
  - 统一目录
  - 统一默认轨
  - 保留 `ASS`
  - 必要时额外生成兼容字幕
  - 只做必要的离线预处理

一句话版本：

`不要把树莓派方案设计成“小型在线视频转码站”，而要把它设计成“自动化番剧整理机 + 直放型媒体库”。`
