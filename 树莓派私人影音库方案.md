# 树莓派私人影音库方案

更新时间：2026-04-03

## 1. 目标

- 做一套纯自用的番剧影音库。
- 能在不同网络环境下访问。
- 以订阅、自动下载、自动整理、前端播放为主。
- 主要片源是每周更新一集的季度番。
- 播放目标以 `Direct Play` 为主，尽量避免边看边转码。

## 2. 限制与边界

### 2.1 当前硬件

- `Raspberry Pi 4B 4GB`
- `32GB SD 卡`
- 现有额外存储为 `SanDisk 1TB USB 3.2 外置 SSD`

### 2.2 使用边界

- 纯自用，不考虑公开分享。
- 本地资源按季度清理，不长期囤片。
- 单独维护“季度追番库”和“收藏库”两套目录。
- `物语` 这类收藏型大目录不纳入自动订阅和自动入库范围。
- 带大量特典、PV、NCOP、扫描图、字体包的大包不默认自动入主库。

### 2.3 性能边界

- Pi 4 适合做下载、整理、刮削、轻量服务、离线批处理。
- Pi 4 不适合承担持续实时视频转码。
- 离线处理优先级应是：
  - 重命名
  - 重封装
  - 统一默认轨
  - 必要时生成兼容字幕
  - 最后才是少量针对性转码

## 3. 已确定

### 3.1 业务方向

- 核心场景是季度追番，不是综合影视库。
- 订阅链路优先围绕番剧设计。
- Jellyfin 作为最终展示和播放前端。
- Tailscale 作为跨网络访问方案。

### 3.2 存储方案

- 不使用 HDD。
- 当前这块 `1TB SanDisk 外置 SSD` 容量充足。
- 系统可以继续放在 `SD` 卡上。
- 不保留原始下载文件，只保留入库后的整理结果。
- 双分区方案可行：
  - `300GB exFAT`：收藏库和与本项目无关的个人文件
  - 剩余空间 `ext4`：下载、处理、Jellyfin 数据、季度追番库、Docker 可写数据
- 但所有高写入目录应放到外置 SSD：
  - 下载目录
  - Jellyfin 数据目录
  - 缓存目录
  - 日志目录
  - 后处理程序数据库
  - Docker 可写卷

磁盘格式建议：

- 如果只做一个分区，优先 `exFAT`，兼容 Mac 和树莓派读写。
- 但 `exFAT` 不适合承载高写入应用数据、下载临时文件和数据库。
- 当前推荐方案就是两个分区：
  - `exFAT`：收藏库，方便 Mac 直接读写。
  - `ext4`：下载、处理、Jellyfin 数据、应用数据。
- 如果这块盘里已经有要保留的收藏文件，分区或重格式化前必须先备份，因为会清空现有数据。

### 3.3 片源特征

当前样本主要集中在这些来源和模式：

- `ANi / Baha / 1080P / AVC / AAC / mp4`
- `Feibanyama / 2160p / HEVC / OPUS / Multi-Subs / mkv`
- `LoliHouse / 1080p / HEVC-10bit / AAC / SRTx2 / mkv`
- `SweetSub&VCB-Studio / mkv + 外挂 ass + 字体包`
- `Dynamis One / 1080p / AVC / AAC / mp4`
- 少量内嵌字幕版
- 少量剧场版和大包资源

这意味着第一版规则不需要追求万能，只要覆盖这几类即可。

### 3.4 字幕与格式处理原则

- 保留原始 `ASS` 字幕，不默认转 `SRT`。
- 不默认烧录字幕。
- 不默认全量转码。
- 默认拒绝 `2160p` 订阅。
- `mp4` 不是天然问题，只有在字幕、音轨、容器兼容性需要时才转 `mkv`。
- 入库前统一命名和目录结构。
- 第一阶段优先依赖 AutoBangumi 的自动重命名能力，不单独维护番名映射表。

兼容版文件说明：

- 兼容版不是替换原片，而是给少数兼容性差的设备额外准备一个更容易直放的版本。
- 例子：
  - 原文件：`HEVC 10-bit + OPUS + ASS` 的 `mkv`
  - 兼容版：`H.264 8-bit + AAC + SRT` 的 `mp4` 或 `mkv`
- 这个功能只在某类设备反复触发转码时再启用，默认不做。

## 4. 仍需明确

- 是否要对部分设备额外生成“兼容版”文件。
- 两套目录在 Jellyfin 中是建成两个独立媒体库，还是先只建一个主库。
- 是否需要在阶段 4 之前就启用兼容版策略。

## 5. 组件职责

### 5.1 AutoBangumi

- 管理 RSS 订阅。
- 匹配番剧规则。
- 把下载任务交给下载器。
- 作为第一阶段的默认重命名器使用。

### 5.2 qBittorrent-nox

- 负责下载。
- 负责下载完成回调。
- 不直接面向最终媒体库。

### 5.3 后处理程序

这是整个方案的核心。

职责：

- 在 AutoBangumi 重命名结果基础上做二次校验
- 探测媒体流信息
- 按规则判断是否可自动入库
- 重命名
- 重封装
- 调整默认音轨和字幕轨
- 必要时生成兼容字幕
- 过滤大包、特典和收藏型资源
- 最终发布到媒体库

### 5.4 Jellyfin

- 只读取“已经处理完成”的媒体库目录。
- 负责展示、元数据和播放。
- 不直接扫下载目录。

### 5.5 Tailscale

- 提供外部网络访问。
- 不暴露下载器和管理面板到公网。

## 6. 最小可行规则

### 6.1 订阅过滤

- 默认排除 `物语` 这类收藏型系列。
- 默认拒绝 `2160p`。
- 默认排除：
  - `PV`
  - `NCOP`
  - `NCED`
  - `menu`
  - `Scans`
  - `CDs`
  - `Fonts`
  - `特典`
- 默认不自动接收结构复杂的大包。

### 6.2 来源策略

- `ANi / Dynamis One`：优先，通常只重命名。
- `LoliHouse`：优先，保守处理。
- `SweetSub&VCB-Studio`：保留 `ASS` 和字体，不做激进兼容化。
- `Feibanyama`：默认降优先级，尤其是 `2160p` 版本。
- 内嵌字幕版：只作为补位源，不作为长期优先标准。

### 6.3 入库规则

- 能识别番名和集号，才允许自动入库。
- 识别失败进入 `manual_review`。
- 只有正片文件才发布到主媒体库。
- 下载完成后源文件不长期保留。
- 大包、特典、菜单、扫描图进入归档或人工处理区。

## 7. 推荐目录

```text
/srv/media
├── downloads
├── processing
│   ├── incoming
│   ├── working
│   ├── failed
│   └── manual_review
├── library
│   ├── seasonal
│   └── collection
└── cache
```

原则：

- 下载区和媒体库必须分离。
- Jellyfin 读取 `library/seasonal` 和 `library/collection`。
- 所有工作目录都放外置 SSD。

## 8. 后处理程序建议

推荐技术栈：

- `Python`
- `SQLite`
- `ffprobe`
- `ffmpeg`
- `mkvmerge`
- `mkvpropedit`

最小模块：

- `parser`
- `probe`
- `decision_engine`
- `executor`
- `publisher`

最小状态：

- `downloaded`
- `planned`
- `processing`
- `published`
- `manual_review`
- `failed`

## 9. 分阶段落地

### 阶段 1：基础服务上线

目标：

- 能下载
- 能播放
- 能通过 Tailscale 访问

范围：

- Raspberry Pi OS
- 外置 SSD 挂载
- Docker Compose
- qBittorrent-nox
- Jellyfin
- Tailscale

具体要做：

- 确认当前系统继续使用 `Raspberry Pi OS 64-bit Trixie`，不立即重装。
- 把外置 SSD 规划为：
  - `300GB exFAT`
  - 剩余空间 `ext4`
- 在树莓派上挂载 `ext4` 分区，例如：
  - `/srv/media`
  - `/srv/appdata`
- 在 Mac 上保留 `exFAT` 分区给收藏和个人文件。
- 安装 Docker Engine 和 Docker Compose。
- 启动 `Jellyfin`、`qBittorrent-nox`、`Tailscale` 基础容器。
- 先不接 AutoBangumi，不做自动订阅。
- 先手工放入 2 到 3 个测试样本，验证 Jellyfin 播放链路。

检查点：

- 树莓派重启后 `ext4` 分区能自动挂载。
- Docker 容器能自启动。
- Tailscale 能从外网设备访问树莓派。
- Jellyfin 能正常扫到测试文件。
- 至少一部 `mp4` 和一部 `mkv` 能在你的主力客户端上直放。
- qBittorrent 下载目录已经落在外置 SSD，不在 SD 卡上。

完成标准：

- 你可以在外网通过 Tailscale 打开 Jellyfin。
- 你可以在 Jellyfin 中播放手工放入的测试番剧。
- 基础服务都已经不依赖 SD 卡承载业务写入。

### 阶段 2：订阅链路上线

目标：

- 能自动订阅季度番
- 能自动提交下载任务

范围：

- AutoBangumi
- 基础订阅规则
- 基础过滤规则

具体要做：

- 部署 AutoBangumi。
- 接通 AutoBangumi 到 qBittorrent。
- 建立第一版季度番订阅规则。
- 写入基础过滤条件：
  - 拒绝 `2160p`
  - 拒绝 `物语`
  - 拒绝 `PV`
  - 拒绝 `NCOP`
  - 拒绝 `NCED`
  - 拒绝 `menu`
  - 拒绝 `Scans`
  - 拒绝 `CDs`
  - 拒绝 `Fonts`
  - 拒绝 `特典`
- 把 AutoBangumi 自动重命名打开，作为第一版标准命名来源。
- 人工选 3 到 5 部在追番剧做试运行。

检查点：

- 新 RSS 条目能自动生成下载任务。
- 过滤规则能挡住明显不需要的资源。
- AutoBangumi 重命名结果已经接近最终库命名。
- qBittorrent 完成下载后文件落在预期目录。
- 没有出现把收藏型资源自动拉进季度追番流程的情况。

完成标准：

- 一周内新增番剧更新可以自动下载。
- 自动下载结果基本符合你的清晰度和来源偏好。
- 过滤规则已经能挡住最常见的大包和非正片资源。

### 阶段 3：后处理程序 MVP

目标：

- 下载后自动规范化入库
- 能挡住大部分脏数据

范围：

- 读取 AutoBangumi 重命名结果
- ffprobe 探测
- 自动重命名
- `manual_review` 队列
- Jellyfin 刷库

具体要做：

- 写一个最小后处理程序。
- 输入来自 qBittorrent 下载完成目录。
- 基于 AutoBangumi 的重命名结果做二次校验。
- 使用 `ffprobe` 读取：
  - 容器
  - 视频编码
  - 音频编码
  - 字幕轨
  - 附件字体
- 做第一版决策：
  - 正片 -> 入 `seasonal`
  - 收藏资源 -> 入 `collection` 或直接忽略
  - 识别失败 -> 进 `manual_review`
  - 非正片/大包 -> 进人工处理区
- 入库完成后刷新 Jellyfin。
- 下载源文件在成功入库后删除，不做长期保留。

检查点：

- 下载完成后能自动触发后处理。
- 至少覆盖 `ANi`、`LoliHouse`、`SweetSub&VCB-Studio`、`Dynamis One` 这几类样本。
- 识别失败的文件不会误入主库。
- Jellyfin 能在不手工干预的情况下看到新集数。
- 成功入库后原始下载文件被清理。

完成标准：

- 从“自动下载完成”到“Jellyfin 可见”这条链路已经自动跑通。
- 绝大部分季度番可以不手工干预入库。
- 失败样本都能进入 `manual_review`，不会污染主媒体库。

### 阶段 4：格式与字幕兼容增强

目标：

- 降低播放时的转码概率

范围：

- 默认轨修正
- 必要时 remux 到 `mkv`
- 兼容字幕策略
- 少量针对性离线处理

具体要做：

- 对播放器兼容性差的案例做复盘。
- 补上默认音轨和默认字幕轨修正。
- 对确实有必要的 `mp4` / `mkv` 做 remux 统一。
- 如确实发现某类设备反复触发转码，再引入“兼容版文件”：
  - 原片保留
  - 单独生成兼容副本
- 兼容版只针对少数固定问题：
  - `HEVC 10-bit`
  - `OPUS`
  - `ASS` 导致的客户端兼容问题
- 不做全量重编码。

检查点：

- 你常用的 2 到 3 类客户端播放时明显更少触发转码。
- 默认字幕轨和默认音轨符合预期。
- 没有因为兼容策略破坏原始字幕体验。
- 兼容版数量可控，没有无节制膨胀。

完成标准：

- 大多数播放请求都能直放或极轻量处理。
- 只有少量特例需要兼容版。
- 兼容版策略已经证明“收益大于复杂度”。

### 阶段 5：长期维护

目标：

- 稳定运行
- 规则逐步收敛

范围：

- 番名映射表维护
- 失败任务复盘
- 订阅规则调优
- 季度清理和归档

具体要做：

- 每周检查 `manual_review` 和 `failed`。
- 把反复出现的问题提炼成规则。
- 调整来源优先级和过滤规则。
- 每季度清理 `seasonal` 目录。
- 收藏内容手工转入 `collection`。
- 如果 AutoBangumi 重命名已经覆盖大多数场景，则继续不单独维护映射表。
- 只有当某些系列长期识别不稳定时，才引入小规模映射表。

检查点：

- `manual_review` 队列规模稳定，不持续堆积。
- 季度切换时不会因为旧数据残留让库变乱。
- 订阅源误判率逐步下降。
- SD 卡使用量和写入压力保持可控。

完成标准：

- 规则趋于稳定。
- 日常维护只需要少量人工介入。
- 季度换番时只需要做规则调整和清理，不需要重构系统。
